Prefetch vs Prerender — Side by Side

Prefetch (this page)

  • 1Rules parsed → browser fetches HTML
  • 2HTML stored in prefetch cache
  • 3You click the link
  • 4Browser serves HTML from cache (0 bytes transferred)
  • 5Normal parse + render still runs
  • 6Page becomes visible

Saves: 1 network round-trip

Prerender (Products/Technology)

  • 1Rules parsed → browser fetches HTML
  • 2Full parse + CSS + JS runs in hidden tab
  • 3Composited frame held in memory
  • 4You click the link
  • 5Browser activates the pre-rendered page
  • 6Page visible instantly

Saves: network + parse + render time

Rule Sources

Beyond the "list" source used in this demo, the Speculation Rules API supports other source types for dynamic or link-based speculation:

JSON — Different source types 
{
  "prerender": [
    {
      // "list" — explicit URL array (used in this demo)
      "source": "list",
      "urls": ["/products", "/checkout"]
    },
    {
      // "document" — speculate on matching links IN the page
      "source": "document",
      "where": {
        // Prerender any same-origin link in .main-content
        "and": [
          { "href_matches": "/*" },
          { "selector_matches": ".main-content a" }
        ]
      },
      "eagerness": "moderate"
    }
  ]
}
Real World Impact

Google's research shows that improving page load time directly impacts user retention and conversion:

→ A 100ms improvement in load time can increase conversion rates by ~1%
→ Prerender can bring Largest Contentful Paint (LCP) to near-zero
→ The API is completely progressive — browsers without support simply ignore the script tag
